## CRITICAL Findings

### C-1: `approveSpender()` bypasses all validator signature checks

**File:** `contracts/src/TradingVault.sol:733-741`
**PoC:** `contracts/test/Adversarial.t.sol::test_ATTACK_approveSpenderDrainsVault` — **PASSES**

A single operator can call `approveSpender(depositAsset, attacker, MAX)` with zero validator signatures, then the attacker drains all vault funds via `transferFrom`. This completely bypasses the m-of-n EIP-712 validation that governs `execute()`.

```bash
# PoC result: vault balance drops from 10,000 to 0
forge test --match-test test_ATTACK_approveSpenderDrainsVault -vv
# [PASS] (gas: 80936)
```

**Impact:** Total loss of all depositor funds. Any operator can drain any vault.
**Fix:** Remove `approveSpender()` entirely. Use `executeWithApprovals()` which ties approvals to validated trades with m-of-n signatures.

### C-2: `unwind()` permits theft of non-deposit held tokens

**File:** `contracts/src/TradingVault.sol:599-621`
**PoC:** `contracts/test/Adversarial.t.sol::test_ATTACK_unwindStealsNonDepositTokens` — **PASSES** (proves attack surface)

During wind-down, `unwind()` is permissionless. It checks that the vault's **deposit asset** balance doesn't decrease, but non-deposit tokens (tokenB, tokenC, etc.) have no protection. An attacker can craft calldata to a whitelisted DEX target that swaps all non-deposit tokens out of the vault.

**Impact:** Loss of all non-deposit-asset positions during wind-down.
**Fix:** Either (a) add balance checks for ALL held tokens, or (b) restrict `unwind()` to `OPERATOR_ROLE` during wind-down.

## HIGH Findings

### H-1: Execute endpoint trusts client-supplied validation without re-verifying signatures

**File:** `trading-http-api/src/routes/execute.rs:244-248`

The `/execute` endpoint accepts `approved: true` and validator signatures from the client (AI agent sidecar) but **never calls** `verify_all_signatures()` from `signature_verify.rs`. A compromised sidecar can submit fake signatures. The on-chain contract catches this for vault trades, but paper trades and CLOB trades have NO signature verification anywhere.

**Impact:** Paper/CLOB trades execute without valid validator approval if the sidecar is compromised.
**Fix:** Call `verify_all_signatures()` in the execute handler before submitting.

### H-2: SSRF via user-controlled `rpc_url` in CLOB approve endpoint

**File:** `trading-http-api/src/routes/clob.rs:55,135`

The `rpc_url` query parameter is attacker-controlled and passed directly to create an HTTP provider. An authenticated agent can scan internal networks or access cloud metadata endpoints (169.254.169.254).

**Fix:** Validate `rpc_url` against an allowlist of permitted RPC endpoints.

### H-3: On-chain `intentHash` not bound to actual trade parameters

**File:** `contracts/src/TradingVault.sol:385-401`
**PoC:** `contracts/test/Adversarial.t.sol::test_ATTACK_intentHashTradeSubstitution` — **PASSES**

Validators sign `(intentHash, vault, score, deadline)` but NOT `target`, `data`, or `value`. An operator can get validators to sign for one trade, then submit different parameters. The on-chain validation passes because `intentHash` is opaque.

**Fix:** Include `target` and `keccak256(data)` in the EIP-712 struct hash.

### H-4: `returnCollateral()` credits `msg.sender`, not releasing operator

**File:** `contracts/src/TradingVault.sol:513-528`
**PoC:** `contracts/test/Adversarial.t.sol::test_ATTACK_returnCollateralCreditsMsgSender` — **PASSES**

When a third party returns collateral on behalf of an operator, the operator's outstanding balance is never reduced. `totalOutstandingCollateral` remains permanently inflated, corrupting `totalAssets()` and share price.

**Fix:** Add an `operator` parameter to `returnCollateral()` to specify which operator to credit.

### H-5: `positionsValue()` sums raw balances without decimal normalization

**File:** `contracts/src/TradingVault.sol:772-776`

A 6-decimal token (USDC) balance of 1M USDC (1e12) is treated as 1e12 wei of an 18-decimal token (0.000001 ETH). This corrupts share price via `totalAssets()`.

**Fix:** Normalize token balances to asset decimals in `positionsValue()`.

### H-6: `adminUnwind()` has no safety limit with default config

**File:** `contracts/src/TradingVault.sol:631-659`

`adminUnwindMaxDrawdownBps` defaults to 0, which skips the drawdown check. CREATOR_ROLE can drain the vault during wind-down.

**Fix:** Set a non-zero default (e.g., 500 = 5%) or require explicit admin opt-in.

### H-7: Bot lifecycle races — no state machine guard

**Files:** `trading-blueprint-lib/src/jobs/*.rs`, `trading-blueprint-lib/src/state.rs`

No mutex or compare-and-swap protects bot lifecycle transitions. Concurrent provision/deprovision, start/stop, or double-activation can corrupt state. The `PersistentStore` lock is released between read and write operations.

**Fix:** Add per-bot `tokio::sync::Mutex` keyed by bot_id for all lifecycle operations.

### H-8: Concurrent workflow ticks — no exclusion mechanism

**File:** `trading-blueprint-lib/src/jobs/workflow_tick.rs`

No mechanism prevents two cron ticks from firing simultaneously for the same bot. Both could propose the same trade, exceeding position limits.

**Fix:** Check "already running" guard in the cron path, not just the HTTP handler.
