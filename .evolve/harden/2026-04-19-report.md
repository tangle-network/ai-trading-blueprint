## Fixes shipped this round

### CRITICAL — fixed

**C1-REG: `approveSpender` regression — still OPERATOR_ROLE instead of DEFAULT_ADMIN_ROLE**
- File: `contracts/src/TradingVault.sol:747`
- Prior harden round claimed this was fixed to DEFAULT_ADMIN_ROLE, but code showed OPERATOR_ROLE.
- A malicious operator could drain the vault via `approveSpender(token, attackerEOA, MAX) + transferFrom`.
- Fix: Changed `onlyRole(OPERATOR_ROLE)` → `onlyRole(DEFAULT_ADMIN_ROLE)`.
- Tests: Updated `TradingVault.t.sol`, `Adversarial.t.sol`, added `AdversarialFuzz.t.sol:test_approveSpenderRequiresAdmin` + `testFuzz_approveSpenderOnlyAdmin`.

**SSRF-1: `rpc_url` from on-chain provision — no URL validation**
- File: `trading-blueprint-lib/src/jobs/provision.rs:162-166`
- Any user calling the on-chain provision job could set `rpc_url` to `http://169.254.169.254/latest/meta-data/` and the operator would issue HTTP requests to the AWS metadata service, leaking cloud credentials.
- Fix: Added `trading_runtime::url_validation::validate_rpc_url()` — blocks internal IPs, metadata endpoints, private networks, non-HTTP schemes, IPv4-mapped IPv6, Docker/K8s internal hostnames. Wired into provision path.
- Tests: 10 new tests in `url_validation.rs` covering loopback, metadata, private networks, IPv6, schemes, CGNAT.

**SSRF-3: `rpc_url` query param on `/clob/approve` — no URL validation**
- File: `trading-http-api/src/routes/clob.rs:135`
- Authed API caller could send arbitrary URL via query param, causing alloy to POST JSON-RPC to internal addresses.
- Fix: Added `validate_rpc_url()` check after URL resolution.

### HIGH — fixed

**RACE-1: Provision TOCTOU — duplicate bot creation**
- File: `trading-blueprint-lib/src/jobs/provision.rs:104-125`
- `find_bot_by_call` check and `bot_store.insert` separated by async container creation (~1-5s). 20 concurrent provisions → 20 bots.
- Fix: Added `PROVISION_INFLIGHT` keyed lock set with `InflightGuard` (drop-safe cleanup on all exit paths). Concurrent provision for same `(service_id, call_id)` rejected immediately.

**RACE-2: `call_id` seconds-granularity collision**
- File: `trading-blueprint-bin/src/operator_api.rs:763`
- `chrono::Utc::now().timestamp() as u64` — two requests in the same second get same call_id.
- Fix: Changed to `timestamp_millis() * 1000 + atomic_counter % 1000`. Sub-millisecond uniqueness.

**RACE-5: Collateral release missing intent dedup**
- File: `trading-http-api/src/routes/collateral.rs:62-114`
- `release_collateral` and `release_collateral_multi_bot` accepted `intent_hash` but never checked it against the dedup store. Concurrent or replayed requests could submit duplicate on-chain transactions.
- Fix: Wired `check_and_insert_intent()` (made `pub(crate)`) into both release handlers. Returns 409 Conflict on duplicate.

### HIGH — fixed (wave 2)

**CHAIN-B / F-8: positionsValue decimal mismatch inflates NAV**
- File: `contracts/src/TradingVault.sol:803-830`
- `positionsValue()` summed raw ERC-20 balances without decimal normalization. A vault holding USDC (6 dec) + WETH (18 dec) had massively inflated NAV.
- Fix: `positionsValue()` now reads `IERC20Metadata.decimals()` for each held token and normalizes to the deposit asset's decimals. Tokens with more decimals are divided, fewer are multiplied. Fallback: raw balance if decimals() reverts.

**SSRF-2: Validator endpoints from on-chain metadata**
- File: `trading-blueprint-lib/src/discovery.rs:34-55`
- On-chain discovery returns user-controlled operator metadata URLs. Once wired, operators could register `http://169.254.169.254/` as their validator endpoint.
- Fix: All discovered endpoints now pass through `validate_rpc_url()`. Internal/metadata addresses are logged and dropped.

**RACE-3: Concurrent activation — double workflow creation**
- File: `trading-blueprint-lib/src/jobs/activate.rs:68-118`
- Two concurrent `activate_bot_with_secrets` calls saw `has_user_secrets=false` simultaneously, both proceeded to create workflows.
- Fix: Added `BOT_LIFECYCLE_LOCKS` — per-bot `tokio::Mutex` acquired at the start of `activate_bot_with_secrets()` and `wipe_bot_secrets()`. Only one lifecycle operation per bot at a time.

### HIGH — deferred (requires protocol version bump)

**CHAIN-A: Calldata not in EIP-712 signed scope**
- Files: `intent.rs:127`, `TradeValidator.sol:215`
- Validators sign `intentHash` which hashes human-readable fields but NOT the actual `target` address or `calldata`. A compromised operator can get valid signatures for a benign-looking intent, then submit different calldata to a whitelisted target.
- Mitigated by: PolicyEngine target whitelist, off-chain `verify_signatures_offchain()`.
- Fix requires: Protocol version bump adding `target` + `calldataHash` to VALIDATION_TYPEHASH (breaking change). Tracked for next protocol version.
